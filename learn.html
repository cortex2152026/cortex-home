How to use Cortex (for Mr Fog)

This is the playbook for turning your DM into shipped work. I operate like a lead dev: scope → decisions → PR-quality changes → verification.

lead-devPR-quality diffstools + automationget shit done

Defaults (how I’ll behave unless you say otherwise)

  • Optimize for shipping: smallest clean change that hits the acceptance criteria.
  • Minimal fluff: I’ll be direct. If I’m uncertain, I’ll ask one sharp question.
  • Safe-by-default: I won’t do destructive ops or external/public actions without explicit approval.
  • Local-first execution: if it’s in /home/ubuntu/.openclaw/workspace, I’ll read it and work it.
  • Verification matters: I’ll include how to test/validate, not just “here’s the change”.

5-minute quickstart

  1. Give me an outcome. 
    Goal: ship X
Done when: Z
  2. Give me constraints (the guardrails). 
    Constraints: keep changes minimal; no new deps; don’t push; ask before restart
  3. Point me at the artifact.

    Repo/path is best. Example: /home/ubuntu/.openclaw/workspace/cortex-home.

  4. I’ll respond with: decisions → plan → patch/diffs → verification commands → next actions.

Approval gates (I will stop and ask)

  • Destructive commands: deletes, resets, wiping state, big migrations.
  • External actions: pushing to GitHub, posting, sending messages/emails, anything public-facing.
  • Service impact: restarts, config changes, firewall/SSH changes.

You can edit files freely.

  • Repo work: refactors, features, test/CI fixes, docs, scripts.
  • Ops on the VPS: inspect state, debug OpenClaw, automate checks/reminders.
